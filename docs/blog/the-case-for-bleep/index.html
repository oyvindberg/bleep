The case for bleep

I have read the recent discussions about sbt 2 and also John de Goes' call for a new Scala build tool with great interest.

I have worked with Scala professionally for a decade, and have strong feelings about the current build tools.

So great in fact, that a year ago I started writing my own (but built on existing, strong foundations!)

Despite people poking fun at the number of failed Scala build tools, I think it's realistic for many projects to replace sbt. The key is seamless migration - more on that later.

Note that it's still early days for bleep, so if the ideas resonate with you, please get involved!

First, let's talk about Bloop​

I see Bloop as a great step forward for Scala tooling.

  • it has great integration with build tools and IDEs through BSP (Build Server Protocol)
  • it provides very good performance through a long-lived, warm JVM
  • it can deduplicate effort between build tools

The Bloop model​

More importantly through, Bloop proves that all Scala builds are simple! Don't believe me? All the builds I've tried can be exported to a bunch of static json files! Then you can compile them, test them, run them, debug them, package them and more.

This proves to me that the the Bloop model is good enough to form the basis of a build tool. The only problem is that the files are incredibly verbose, and not portable.

Let's see where this leads us...

Towards a non-verbose Bloop model​

Deduplicate with templates:

-projects:
-  kernel:
-    platform:
-      name: jvm
-  core:
-   dependsOn: kernel
-   platform:
-     name: jvm
+projects:
+  kernel:
+    extends: common
+  core:
+    dependsOn: kernel
+    extends: common
+templates:
+  common:
+    platform:
+      name: jvm

Built-in cross-compilation​

projects:
  core:
    dependencies: com.olvind.tui::tui:0.0.5
    cross:
      jvm3:
        scala:
          version: 3.0.0
      jvm213:
        scala:
          version: 2.13.0
    platform:
      name: jvm

In fact anything can vary, and you can use any names. These are just variations of the same project

projects:
  core:
    cross:
      ce2:
        dependencies: org.typelevel::cats-effect:jar:2.5.5
      ce3:
        dependencies: org.typelevel::cats-effect:jar:3.4.8
    scala:
      version: 2.13.0
    platform:
      name: jvm

The project structure is the same at the projects, cross and template levels. Projects are treated as sets of data, and settings from the different levels are merged with reliable semantics in case of conflicts.

Build as data​

I agree that build tools should be pure data. This way you can load the build instantly, you can merge, rewrite, patch builds with ease.

By adopting the portable, non-verbose Bloop model, "build as data" is the core of bleep.

When build as data is not enough​

I get it. You need to, say, upper-case all class file names before you package them. Or something like that. In any case, code as data is not enough.

In my mind there is a solution which is so simple and so elegant. You write code! You absolutely do not embed that code in your build, but you let code load the build. As data of course!

Enter scripts:

projects:
  my-scripts:
    platform:
      name: jvm
scripts:
  uppercase-and-package:
    main: mypackage.UppercaseAndPackage
    project: my-scripts

mypackage.UppercaseAndPackage is a completely normal Java or Scala class with completely normal main method. Which means you can use your IDE to write it, debug it, test it, and so on.

Optionally you can include bleep code to load and manipulate the build, as a completely normal dependency in a completely normal project.

projects:
  my-scripts:
    dependencies: build.bleep::bleep-core:${BLEEP_VERSION}

If you want you can even do transformations of the build, and load that instead. Here is an example of a script which upper-cases all project names.

object CompileUppercaseProjects {
  val rewrite = new BuildRewrite {
    override val name = BuildRewriteName("uppercase-projects")

    override protected def newExplodedProjects(oldBuild: Build): Map[CrossProjectName, Project] =
      oldBuild.explodedProjects.map { case (k, v) => uppercase(k) -> v.copy(dependsOn = v.dependsOn.map(uppercase)) }
  }

  def main(args: Array[String]): Unit = {
    val (commonOpts, _) = CommonOpts.parse(args.toList)

    bootstrap.forScript(scriptName = "compile-uppercase-projects", commonOpts, List(rewrite)) { (started, commands) =>
      commands.compile(List(started.build.explodedProjects.keys.last))
    }
  }
}

You can obviously write that build back to file: (though the output is less pretty than the input, for now)

bleep.yaml.writeShortened(
  started.build.requireFileBacked(scriptName).file,
  started.buildPaths.bleepYamlFile
)

This is where the model is at now, and I think I can leave it to your imagination to see where it can (and can not) go.

  • Share templates for all your projects? Sure, let's extend bleep with some download/merge build functionality
  • Share scripts? Sure, publish them as library, use git submodules, or whatever you want
  • Write a script to bump a dependency across all your builds? Sure, hack up a script which loads the yaml files and writes them back

Developer experience​

The model wasn't necessarily the reason why I started bleep, developer experience (DX) was!

Performance, performance, performance​

The build tool needs to load instantly. Milliseconds. Project reloads should be instant. Tab-completion should be instant. Everything should be instant.

bleep is fast not by implementation (it has not been optimized, at all), but by design:

  • It is compiled with GraalVM native-image
  • The loading of the build is just parsing a yaml file, and then "exploding" the projects into their full form.
# loading a small build
$ time bleep projects>/dev/null
bleep projects > /dev/null  0.01s user 0.02s system 34% cpu 0.094 total

# loading a huge build (tapir)
$ time bleep projects>/dev/null
bleep projects > /dev/null  0.03s user 0.03s system 80% cpu 0.067 total

There are two things which can be slow in bleep:​

Resolving dependencies​

To mitigate this, the "dependency language" has shed some ivyisms for cacheability. And it is cached to disk, of course

Starting a JVM​

The idea is that you pay for what you use. Switching from one branch to another? Instant project reload!

Running the script to build your docker image after upper-casing the class file names? You're paying the JVM startup cost.

The important thing is that all the most frequent tasks are fast!

Usability​

There are so many things here. For instance:

Creating a build should be trivial​

Running bleep in a directory gives you the option of importing from sbt or creating a new build.

File system matters​

You can cd into subdirectories to work on a subset of your build

$ bleep projects|wc -l
565
$ cd json
$ bleep projects|wc -l
90

Tab completion​

bleep instantly tab-completes all commands, projects, scripts and options for bash and zsh (so far)

You can easily mount a subset of projects into your IDE​

bleep setup-ide js3 # picks all projects which have a `js3` cross

IDE reloads automatically​

This really only works in metals right now, but it is a huge DX improvement.

Migration strategy​

Nobody sits down and rewrites their builds manually. This is why Mill, even though it is vastly better than Sbt, has not gained enough traction.

Bleep on the other hand can piggyback on the existing bloop export from sbt! sbt bloopInstall generates a bunch of json files, bleep can read those and generate a bleep yaml file.

Here is a demo where zio-http is downloaded, imported and tested with bleep.

As you can see it provides one-command import, so you can test bleep right away. If you like it, there is some more work to do:

  • the import generates makeshift resource generation scripts if your sbt build produces any. These are completely static, so you'll need to write code and use plugins to make them dynamic again.

  • you need to write scripts for your own use cases.

For instance, in the bleep repo there are scripts for:

  • publish to maven central
  • build native-images
  • generate video
  • generate documentation
