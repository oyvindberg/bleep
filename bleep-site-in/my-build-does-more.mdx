# But my build does much more!

Yes, your build does other things. It generates code, publishes artifacts, builds websites, runs linters, spins up integration tests. Probably a lot more.

None of this has to live inside the build tool itself.

Bleep handles everything beyond `compile` / `test` / `run` with **scripts** — ordinary Scala programs that you write, debug, and run like any other. There is no DSL, no task graph, no macro magic, no plugin classloader — just code.

There are two flavours:

- **[Scripts](/docs/usage/scripts)** — a `main` method wired into the build. Runs with `bleep <script-name>`, has access to the build model, can invoke other bleep commands. Use this for publishing, docs generation, custom CI logic, and anything else that isn't compile/test/run.

- **[Source generation](/docs/usage/sourcegen)** — a script that produces `.scala` / `.java` / resource files consumed by another project. Runs automatically before compilation, with proper invalidation and atomic failure handling. Use this for version strings, protobuf-generated code, OpenAPI clients, and anything else you'd have put in sbt's `sourceGenerators`.

The common thread: **it's all just code.** You can run it from your IDE, debug it with breakpoints, use any dependency from any Scala version, and reason about it the same way you reason about the rest of your codebase.
