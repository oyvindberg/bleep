# Why Bleep?

Bleep is a Scala build tool that prioritizes **speed**, **simplicity**, and **developer experience**.

## Key Advantages

### Instant Startup

Bleep is compiled to native code using GraalVM:

- **~0ms startup time** - No JVM warmup required
- **Responsive CLI** - Every command feels immediate
- **Fast CI pipelines** - No build tool overhead

### Declarative YAML Configuration

Your build is described in a single `bleep.yaml` file:

```yaml
$schema: https://raw.githubusercontent.com/oyvindberg/bleep/master/schema.json
$version: 0.0.14
projects:
  myapp:
    dependencies:
      - com.lihaoyi::fansi:0.5.0
    scala:
      version: 3.3.3
    platform:
      mainClass: com.example.Main
```

Benefits:
- **Readable** - Anyone can understand your build
- **Diffable** - Build changes are clear in code review
- **Machine-manageable** - Tools can modify builds programmatically
- **IDE support** - Schema validation with autocomplete

### One-Second IDE Imports

Bleep uses BSP (Build Server Protocol) for IDE communication:
- **IntelliJ IDEA** and **Metals** import in about one second
- Changes reflect immediately - no re-import wait

## How It Works

Bleep builds on [Bloop](https://scalacenter.github.io/bloop/), the Scala build server that handles fast incremental compilation.

What bleep adds:
- **Portable build format** - YAML instead of machine-bound JSON
- **Dependency resolution** - Via Coursier
- **Native CLI** - Instant startup
- **Scripts** - Custom build logic as regular Scala programs
- **Cross-building** - JVM, JS, and Native support

## Trade-offs

### Less Flexible Than sbt

sbt allows arbitrary Scala code in builds. Bleep uses YAML with scripts for custom logic - simpler but less flexible for unusual requirements.

### Smaller Plugin Ecosystem

sbt has more plugins. Many common operations work in bleep, and you can [port sbt plugins](/docs/porting-sbt-plugins) as needed.
